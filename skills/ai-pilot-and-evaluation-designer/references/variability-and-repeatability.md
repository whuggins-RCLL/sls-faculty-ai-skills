# Variability and repeatability

The same input can produce different outputs across runs, accounts, and dates. A single run is not a result. Test variability where it matters to the decision.

## Capture the configuration

Record for every run:

- Product
- Model or family
- Feature
- Tier
- Region
- Tools enabled
- Retrieval configuration
- System instructions
- Administrative settings
- Test date

## Repeat where it matters

- Rerun selected cases multiple times and record the spread of outcomes.
- Separate stylistic variation (wording) from substantive variation (different conclusions, different citations, different omissions).
- For high-stakes cases, substantive variability across identical inputs is itself a serious finding.

## Report variability honestly

- Do not report a best-of-several run as the result.
- Do not treat repeated outputs from one case as if they were independent users.
- State how many trials were run and how outcomes were distributed.

## When variability is disqualifying

If the workflow requires consistent, defensible outputs (for example, the same legal conclusion for the same facts) and the system produces materially different conclusions across runs, that undermines the use regardless of average quality.

## Handoff

Product and model changes over time are handled in the change-control reference. Isolating why outputs vary can be explored in the Capability and Failure Lab.
