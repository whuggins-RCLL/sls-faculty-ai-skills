# Pilot readiness

Before designing a full pilot, determine whether the following are defined well enough to test.

## Readiness checklist

| Element | Defined? | Notes |
|---|---|---|
| Problem | | The real problem, not the tool |
| Intended users | | Who will actually use it |
| Current workflow | | How the task is done now |
| Baseline | | Measured or describable current performance |
| Narrow candidate task | | One representative task, not everything |
| AI role | | Draft, retrieve, recommend, check, etc. |
| Human responsibilities | | Who verifies, decides, is accountable |
| Exact system or product | | Product, model, feature, tier, configuration |
| Data needs | | What data, what sensitivity |
| Institutional questions | | Data classification, approval, accessibility |
| Test materials | | Public, synthetic, or authorized cases |
| Review method | | How outputs will be judged |
| Decision the results will support | | The concrete choice at the end |

## Readiness findings

Report one calibrated finding, with the reason and the next step:

- Ready for a rapid pilot
- Ready for a structured pilot
- Needs clearer problem definition
- Needs baseline measurement
- Needs product investigation
- Needs a capability test first
- Needs data or institutional review
- Needs workflow redesign
- Poor pilot candidate
- Insufficient information

## Routing

- Incomplete problem framing or "should we use AI at all" → SLS Problem-to-AI Use Case Studio.
- A single unproven behavior that must be seen first → SLS AI Capability and Failure Lab.
- Unknown product facts → SLS AI Product Decoder, then return here.

## Do not

- Do not proceed to a full protocol with an undefined decision.
- Do not treat enthusiasm or a good demo as readiness.
- Do not invent a baseline to unblock the checklist.
