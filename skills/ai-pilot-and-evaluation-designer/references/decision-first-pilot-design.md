# Decision-first pilot design

Design the pilot backward from the decision it must support. This prevents collecting interesting results that do not answer the actual question.

## State the decision first

Every substantial pilot opens with:

### Decision the Pilot Will Support

At the conclusion, the authorized decision-maker should have evidence to decide whether to:

- Stop
- Revise and retest
- Continue limited exploration
- Conduct a larger pilot or evaluation
- Seek institutional review
- Begin deployment planning
- Select among defined alternatives

Name the decision-maker (role, not a real named individual in fictional scenarios) and the realistic options in front of them.

## Reject vague purposes

Do not accept:

- See how it goes
- Explore AI
- Test the tool generally
- See whether faculty like it

Convert each into a concrete decision. "See how it goes" becomes, for example, "Decide whether to continue a limited pilot of this drafting assistant for one seminar's memo feedback next term."

## The pilot question

For [defined users] performing [defined task] with [defined materials], does [exact system and configuration] improve [defined outcome] compared with [baseline or alternative], while keeping [defined errors, omissions, review burden, data exposure, access barriers, and costs] within acceptable limits?

Fill every bracket. If a bracket cannot be filled, that is a readiness gap, not a detail to skip.

## Working backward

1. Decision and options.
2. What evidence would change the decision either way.
3. What outcomes and limits that evidence requires measuring.
4. What test set, baseline, and review method produce that measurement.
5. What thresholds separate the options — set before results.

## Guard

If the pilot cannot, in principle, produce a "stop" or "do not adopt" outcome, it is a demonstration in disguise. A valid pilot must be able to fail.
